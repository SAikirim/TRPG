Illustration System Details
===

Image Generation Method (Skia Immediate + SD Replacement)
===
All image generation operates in 2 stages:

  [Stage 1] Skia Immediate Generation (within 0.1 seconds)
    - Skia immediately generates a placeholder image and displays it on the web
    - Provides instant visual feedback without a blank screen

  [Stage 2] SD Background Generation (10~30 seconds)
    - SD WebUI generates a high-quality image in the background
    - Upon completion, automatically replaces the Skia image with the SD image
    - The web UI's 2-second polling automatically detects the new image
    - If SD is OFF or SD fails, the Skia image remains as the final version

Applicable targets:
| Target | Skia Placeholder | SD Final |
| Scene background | Vector background (forest/dungeon/village, etc.) | SD high-quality background |
| NPC/PC portrait | Silhouette + color | SD portrait |
| World map background | Terrain-colored map | SD fantasy texture |
| Object icon | Vector icon | Skia retained (SD unnecessary) |

If SD is turned off, Skia serves as the final image.

SD Operation Check:
- Controlled by the sd_illustration value in current_session.json
- Verified via is_sd_enabled() in sd_generator.py
- If SD WebUI is unresponsive, the Skia image remains as the final version

Skia Usage Principles:
- Prefer curves (Bezier) over straight lines — for a fantasy feel
- Always enable anti-aliasing
- Vector-based, so resolution can be set freely
- Actively utilize gradients, transparency, and shadows

SD Start/Stop:
- SD WebUI is automatically checked at game session start (see guides/process.txt)
- If sd_illustration is true, SD must be running
- If SD is not running, the Skia placeholder remains as the final image — no impact on gameplay

SD Illustration Integration System (sd_generator.py)
===

SD ON/OFF Toggle:
- Controlled by the sd_illustration value in current_session.json
- Can be toggled via the web UI toggle button or the /api/illustration/toggle API

sd_generator.py Behavior:
- Calls SD WebUI API via async thread (request_illustration)
- Auto model check: automatically switches if not dreamshaper_8
- Generation status tracking: _scene_state["generating"]["status"] (idle/generating/error)
- Thread safety: scene_state protected by threading.Lock

Layer System
===
Visual novel-style layer structure:
  [Background image - full]
    +-- [Character portrait - left/center/right]
    +-- [Object - position specified]

- Existing layers are automatically cleared when generating a background
- Remove specific layers with remove_layer("name")
- Reset everything with clear_scene

Generation Sizes:
| Type                      | Size     |
| Background (background/scene) | 768x512  |
| Portrait (portrait)       | 384x512  |
| Object (object)           | 256x256  |

SD Parameters
===
- Model: dreamshaper_8.safetensors
- Sampler: DPM++ 2M Karras
- Steps: 20, CFG Scale: 7
- Negative: lowres, bad anatomy, bad hands, text, watermark, worst quality, low quality

gm-update Illustration Payload
===

Background generation:
  {"illustration": {"type": "background", "prompt": "..."}}

Character portrait placement:
  {"illustration": {"type": "portrait", "prompt": "...", "position": "left", "name": "Saiki"}}

World Map Generation Rules
===

Background Image Generation:
  Two backgrounds are always generated based on natural data (terrain):
  1. Skia base map (immediate) → background_skia.webp
  2. Color blob guide → SD img2img → background_sd.webp (background)

Background Display Priority:
  Priority 1: custom/background.webp — User-provided base map (used unconditionally if present)
  Priority 2: background_sd.webp — SD-corrected background (when sd_illustration=true)
  Priority 3: background_skia.webp — Skia fallback (when sd_illustration=false or SD not generated)

Natural Data Generation:
  [When user provides an image]
    User base map → terrain_extractor auto-extracts → reflected in worldbuilding.json terrain
  [No user image — new scenario/worldbuilding]
    Worldbuilding Agent: terrain design (continent shape, climate, vegetation concept)
    World Map Agent: coordinate placement + geographic verification (coastlines, mountain ranges, river flows)
    → Saved to worldbuilding.json terrain section
  [When adding a region]
    Reference existing world map (adjacent region boundaries, coastline continuity)
    Consistency ensured via neighbors information

Civilization Placement (after natural data is finalized):
  Worldbuilding Agent + World Map Agent place locations according to terrain:
    Bay/estuary → port, river confluence → trade city, mountain pass → fortress, fertile plains → farming village

Final Map Composition:
  Background (above priority) + marker overlay (locations, roads, labels, current position, compass, title, legend)
  → world_map.png

Folder Structure (static/maps/world/{worldbuilding}/):
  custom/background.webp  — User-provided original (optional)
  background_sd.webp      — SD img2img corrected background
  background_skia.webp    — Skia fallback base map
  color_guide.png         — Color blob guide (SD input)
  world_map.png           — Final completed map (background + markers)

Auto-generation Conditions:
  - When creating a new scenario/worldbuilding (full pipeline)
  - When the active region changes to a different country/region
  - At Flask server startup (markers only recomposed, background cache reused)
  - On gm-update call (markers only recomposed)
  - /api/world-map/regenerate (full forced re-execution)

Cache Policy:
  - User images, SD backgrounds, Skia backgrounds are cache-preserved
  - Color guide + Skia regenerated only on terrain changes
  - world_map.png recomposed only on marker changes
  - Existing cache preserved on worldbuilding switch (different folders)

World Map Post-processing (SD-based Terrain Correction)
===

Full World Map Workflow:
  [1] Write or modify terrain data (worldbuilding.json terrain.features)
  [2] Generate SD base map (generate_world_map or SD WebUI)
  [3] SD-based terrain correction (visually inspect SD output → fine-tune terrain coordinates)
  [4] Regenerate overlay map (world_map.png)
  [5] Final review — Worldbuilding + World Map Agents review in parallel
  [6] GM integrated judgment + corrections
  [7] Data validation (check for contradictions)
  [8] Final confirmation + docs/ sync

Why This Is Needed:
  - SD generates backgrounds based on prompts and terrain data, but cannot accurately reflect coordinates
  - The generated SD image is the "actual map," and terrain data must be corrected to match it
  - Without correction, the world_map.png overlay (location markers, terrain colors) will misalign with the SD background

Correction Targets:
  - Mountain range (mountain) coordinate ranges
  - Forest (forest) coordinate ranges
  - Plains (plains) coordinate ranges
  - River (river) paths
  - Coastline (coastline) paths
  - Sea (sea) coordinate ranges

Correction Method:
  - Open background_sd.webp with the Read tool and visually inspect it
  - Compare the SD image's color regions (brown=mountains, green=forests, light=plains, blue=sea) with current coordinates
  - Adjust coordinates to match the SD image
  - Do not change location (locations) data — correct terrain only
  - After correction, regenerate world_map.png and verify that the overlay matches the background

Auto-trigger Conditions:
  - When a new SD base map has been generated
  - When terrain data in the worldbuilding has been significantly modified
  - When the world_map.png overlay and background_sd.webp appear misaligned at session start

Notes:
  - Skip this step if background_sd.webp does not exist or is a Skia fallback
  - Correction targets only terrain coordinates (terrain.features)
  - Location coordinates (locations.*.world_pos) are changed only through separate judgment
  - Always keep both files (data/worldbuilding.json, worldbuildings/*.json) identical

World Map Final Review (Agent Verification)
===
After generating or modifying a world map, the Worldbuilding Agent and World Map Agent must perform a final review.
The GM makes an integrated judgment on the review results to determine whether corrections are needed.

Trigger Conditions:
- When a new world map has been generated
- When location coordinates (world_pos) have been changed
- When connection distances/directions have been changed
- When terrain data (terrain.features) has been significantly modified
- After performing SD-based terrain correction

Review Process:
  [1] Agent [Worldbuilding] Review (parallel)
    - Geographic rationality of location placement (cities on plains, villages in forests, ports on coasts)
    - Logic of connections/routes (reasons for detours, trade route names matching actual paths)
    - Rationality of distances (route distance shorter than coordinate straight-line distance is physically impossible)
    - Contradictions between terrain and lore
    - Whether overall geography is narratively consistent

  [2] Agent [World Map] Review (parallel)
    - Coordinate and terrain type alignment (port is on coast, not in the sea)
    - Consistency between terrain overlay and SD background
    - River flow direction (highlands → sea)
    - Coastline path matching SD image
    - Connection direction labels matching actual coordinate directions
    - Scale consistency between distances and coordinates
    - Label readability

  [3] GM Integrated Judgment
    - Collect review results from both agents
    - Consider game progress, existing narration, and scenario flow
    - Decide to correct/defer/ignore each issue
    - If corrections contradict existing event logs, also modify the logs
    - After final corrections, regenerate world_map.png and verify the results

  [4] Data Validation After Corrections (contradiction check)
    After the GM applies corrections, verify through automatic/manual validation that no new contradictions exist.
    - Run session_validator.py — automatic validation of game state, NPCs, save consistency
    - Visually compare coordinate data with the final rendered image (world_map.png)
      - Whether markers are on the correct terrain (port on coast, not in the sea)
      - Whether rivers connect unbroken from mountains to sea
      - Whether terrain overlay colors match the SD background
    - Reconfirm connection distances/directions affected by corrections
    - Check that event logs contain no content contradicting pre-correction data
    - If problems found → return to [3] for re-correction
    - If no problems → proceed to [5]

  [5] Final Confirmation
    - Regenerate world_map.png + sync docs/
    - git commit
    - Record verification complete

Review Output Format:
  Each agent classifies issues by severity (HIGH/MEDIUM/LOW) and reports them.
  - HIGH: Physically impossible or obvious contradiction — must be corrected
  - MEDIUM: Insufficient consistency or lore reinforcement needed — correction recommended at GM's discretion
  - LOW: Fine-tuning or visual improvement — low priority

Notes:
- Do not apply agent review results as-is — the GM makes an integrated judgment in the overall context
- Do not finalize the world map without review
- If corrections result from the review, map regeneration + docs sync must be completed

Scenario Pre-generated Images
===
When the Scenario Agent generates a scenario, all required images are pre-generated based on the entire story.
Can be displayed immediately during gameplay without real-time generation wait. Format is WebP.

Pre-generation Targets:
| Category            | Filename Convention              | Size    | Background |
| Chapter background  | background_{name}.webp           | 896x512 | Yes        |
| Location/event bg   | background_{name}.webp           | 896x512 | Yes        |
| NPC/Monster         | portrait_{name}.webp             | 384x512 | Transparent|
| Player character    | portrait_{name}.webp             | 384x512 | Transparent|
| Object              | object_{name}.webp               | 256x256 | Transparent|

Character/Object Transparent Background Rules:
- Characters and objects are generated without background (transparent) -> can be composited on any background
- After SD generation, background is automatically removed using the transparent-background library
- "detailed background, scenery, landscape" is automatically added to the negative prompt
- Including "simple background, solid color background" in the prompt is recommended

Scenario Image Generation Procedure:
1. Scenario Agent understands the entire story when generating scenario.json
2. Extract required image list: all chapter backgrounds, appearing NPCs, key objects, key locations
3. Write English prompts for each item
4. Generate sequentially via SD API (simultaneous generation prohibited due to VRAM constraints)
5. Quality check -> re-generate with improved prompts if issues found
6. Save to static/illustrations/sd/ and static/portraits/sd/
7. git commit + push (docs/ auto-sync)

Default Background Auto-display:
- The web UI automatically displays the background matching the current chapter
- Priority: GM-requested scene > SD pre-generated background > Skia fallback background
- Even when SD is OFF, the Skia background is always displayed (no blank screen)
- The /api/illustration endpoint in app.py provides default_bg (sd/pixel paths)

Illustration Generation Criteria
===
- Do not generate illustrations for insignificant NPCs/creatures (pack horses, background animals, etc.)
- Generate only when explicitly requested by the user
- Use original size for illustration images — do not pre-shrink
- Display appropriately on the web via CSS (controlled with max-width/max-height)

NPC Portrait Generation Requirements
===
- When generating NPC portraits with SD, always reference the appearance field in entities/{scenario_id}/npcs/npc_{id}.json
- Reflect appearance's age, build, skin, hair, face, outfit, notable in the English prompt
- The race field must also be reflected (human/elf/orc, etc.)
- Use clothing/props appropriate to the worldbuilding setting (medieval fantasy) — modern style prohibited
- After generation, verify that the appearance matches the entity settings

Portrait/Object Auto-prompt Generation
===
- When request_illustration(type="portrait", prompt="", name="name") is called with an empty prompt, it is auto-generated
- Objects (type="object") follow the same behavior
- How it works:
  1. Search for entity JSON matching the name in entities/{scenario_id}/npcs/ and players/ directories
  2. If found, convert race + appearance fields to an English SD prompt (_appearance_to_prompt)
  3. Race uses a Korean→English mapping table (인간→human, 엘프→elf, 말→horse, etc.)
  4. Appearance's age, build, skin, hair, face, outfit, notable are added to the prompt in order
  5. Style keywords auto-added: fantasy, semi-realistic, upper body portrait, simple background, masterpiece, best quality
  6. If entity not found, falls back to "fantasy character portrait, {name}, simple background, masterpiece"
- If a prompt is explicitly provided, auto-generation is skipped and the provided prompt is used as-is
- With this feature, even portrait requests with empty prompts generate images matching the character settings

Character Appearance Consistency Rules
===
- The same character must always maintain the same appearance (face, build, outfit, hair color)
- Store a reference image for each character in static/portraits/sd/ or static/portraits/pixel/
- Reuse existing images when the same character appears in a new scene
- Generate new images only when appearance changes (equipment change, injury, etc.)
- When only pose/action needs to change: transform the existing image via img2img
- Layer portraits must have transparent backgrounds — portraits with backgrounds obscure the main background
- Background is automatically removed using the transparent-background library during SD generation
- If background removal fails, log a warning + save as-is (manual review required)

Character Image File Rules:
| Purpose               | Path             | Naming Convention                  | Background    |
| Reference image (front) | portraits/sd/  | portrait_{name}.webp               | Transparent   |
| Action variant        | portraits/sd/    | portrait_{name}_{action}.webp      | Transparent   |
| Profile card          | portraits/pixel/ | player_{id}.png                    | Circular mask |

Examples:
- portrait_Hallan.webp (default — front-facing)
- portrait_Hallan_injured.webp (injured state)
- portrait_Mira.webp (default)
- portrait_Mira_angry.webp (angry expression)

Emoji Rendering Rules
===
- Emojis used in map PNG images must be compatible with Windows seguiemj.ttf
- The safe_emoji() function in map_generator.py automatically handles unsupported emoji fallback
- Unsupported detection criteria: alpha channel pixels < 150 when rendered at 30x30px (no color glyphs)
- New emojis must always go through safe_emoji()
- Fallback mapping: registered in the EMOJI_FALLBACK dictionary
- This restriction does not apply to HTML web UI since the browser handles rendering (PNG maps only)
- The icon field in object entity JSON can retain the original emoji — safe_emoji handles it at render time

Image Quality Management
===
- Quality check after generation is mandatory: check for blurry, out of focus, deformed
- Re-generate problematic images with improved prompts
- Prompt improvement tips:
  - Blurry results: add "blurry, out of focus, bokeh, depth of field" to negative
  - Hand/body distortion: add "bad hands, bad anatomy, deformed, extra fingers" to negative
  - Out of focus: add "sharp focus, 4k" to prompt
  - People included in background: add "people, characters, person" to negative
