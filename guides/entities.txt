Entity/NPC System Details
===

NPC Entity Mandatory Rules
===
When a new NPC appears, an entity file must be created. No exceptions.

Auto-Generation System
===
- NPCs added via gm-update API's new_npcs -> automatically create entities (app.py hook)
- python game_mechanics.py check_npcs -> auto-creates entities for NPCs in game_state that lack one
- ensure_all_npc_entities() called at session start -> automatically fills in missing entities

Cases requiring manual creation:
- When a named NPC newly appears in GM narration
- Call game_mechanics.create_npc_entity() or write JSON directly
- Minimum required fields: id, name, type, status, personality, memory

Entity Structure (JSON Example)
===
All NPCs (including monsters) use the same 4-stat (STR/DEX/INT/CON) + HP/MP system as players.
stats, hp, max_hp, mp, max_mp are required fields. Refer to rules.json's npc_stats.guidelines for stat ranges.

{
  "id": 300,
  "name": "Harlan",
  "type": "friendly",        // friendly | monster | neutral
  "known": false,             // Required — whether name is revealed to players. If false, displayed as "???"
  "status": "alive",          // alive | dead | fled | unconscious | looted | gone | separated
  "location": "Trade Route",
  "stats": {                  // Required -- same 4 stats as players
    "STR": 10,
    "DEX": 10,
    "INT": 14,
    "CON": 10
  },
  "hp": 12,                   // Required -- current HP
  "max_hp": 12,               // Required -- maximum HP
  "mp": 3,                    // Required -- current MP
  "max_mp": 3,                // Required -- maximum MP
  "race": "Human",            // Race (Human, Elf, Dwarf, Orc, Beastfolk, Slime, Golem, Wolf, etc.)
  "appearance": {              // Appearance description — used for SD prompt generation and narration
    "age": "50s",
    "build": "Stocky middle-aged man",
    "skin": "Brown skin",
    "hair": "Short gray-streaked brown hair",
    "face": "Deep wrinkles, sharp eyes",
    "outfit": "Leather vest, cloth pants",
    "notable": "Bandage on left calf"
  },
  "personality": {
    "temperament": "gruff",   // Personality temperament
    "intelligence": "high",   // Intelligence level
    "behavior_pattern": "",   // Behavior pattern description
    "speech_style": "",       // Speech style
    "motivation": ""          // Motivation/purpose
  },
  "relationships": {},        // Relationships with other characters
  "memory": {
    "dialogue_history": [],   // Key dialogue records
    "key_events": []          // Key event memories
  }
}

- Monsters may keep the existing attack, defense fields for convenience, but actual checks are performed based on stats
  - attack = weapon_damage + STR modifier (or INT modifier for magic monsters)
  - defense = 10 + DEX modifier + armor

NPC Relationship System
===
NPCs use a structured relationship format with numeric affinity/trust values and event history.

JSON Structure:
  "relationships": {
    "target_name": {
      "description": "관계 설명 (한국어)",
      "affinity": 65,
      "trust": 70,
      "history": [
        {"turn": 8, "change": 15, "reason": "늑대에서 구해줌"}
      ]
    }
  }

Fields:
- description: Korean text describing the relationship
- affinity: 0-100 emotional closeness scale
- trust: 0-100 reliability/faith scale
- history: Array of events that changed the relationship (turn, change amount, reason)

Affinity/Trust Scale:
- 0-19: hatred / complete distrust (hostile actions, betrayal aftermath)
- 20-39: dislike / wary (negative first impression, past conflict)
- 40-59: neutral (strangers, no strong feelings)
- 60-79: favorable / friendly (positive interactions, rescued, helped)
- 80-100: deep bond / absolute trust (family, lifelong companions, proven loyalty)

When Affinity/Trust Changes:
- Positive: helping in danger (+10~20), kind conversation (+3~5), gift/trade (+5~10), fulfilling a promise (+5~10)
- Negative: insult/rudeness (-5~10), theft/betrayal (-20~40), breaking a promise (-10~15), harming allies (-15~30)
- Trust recovers slower than affinity (trust is harder to rebuild once broken)
- Family/long-term bonds start high but can still change with extreme events

Data Flow: NPC Agent -> relationship_changes in response -> GM Orchestrator -> update entity JSON file
- NPC Agent reads entity file (including relationships) before generating dialogue/actions
- NPC Agent includes relationship_changes array in response when interactions affect relationships
- GM Orchestrator applies relationship_changes to the NPC entity file after each turn
- Format: {"npc_id": 300, "target": "saiki", "affinity_delta": +5, "trust_delta": +3, "reason": "친절한 대화"}

CLI Commands
===
  python game_mechanics.py npcs          # List NPC entities
  python game_mechanics.py check_npcs    # Verify missing + auto-create

Agent Continuity (File-Based Memory)
===
Agent tools are one-shot, so they have no conversation context. Continuity is maintained in two ways:
1. Context provided by GM: this turn's direction, user action, current situation (passed in phase 1b)
2. JSON file-based memory: entity state, memories, relationships, etc.
- On invocation: receive GM context + read JSON to restore -> decide/act -> save JSON -> return result
- Scenario isolation: managed under entities/{scenario_id}/
- World setting is scenario-independent: worldbuilding.json

Agent Execution Backends (Hybrid Architecture)
===
Each agent can run on a different backend. Configured in data/agent_config.json.

Runners:
- claude: Claude Code Agent tool (default). No external API needed.
  - GM builds prompt via core.runners.registry dispatch_claude_prompt()
  - Passes prompt to Agent tool manually
- cloud: External LLM APIs (Gemini Flash, Groq/Llama3, OpenAI)
  - Requires API key in environment (GOOGLE_API_KEY, GROQ_API_KEY, OPENAI_API_KEY)
  - Programmatic execution via core.runners.registry dispatch()
- local: Local models via Ollama API (http://localhost:11434)
  - Requires Ollama running with the specified model

All runners use the same pydantic schemas (core/schemas/) for input/output.
Schema details: AgentRequest wraps TurnContext + agent-specific payload.
Response: AgentResponse wraps success/payload/warnings/errors.

Recommended model assignments:
- Logistical agents (rule, scenario, worldbuilding, worldmap, system): models with strong JSON + long context
- Roleplay agents (npc, player): models with strong dialogue + personality consistency
- When all agents use claude runner: no external dependencies, simplest setup

Agent Status Display Rules
===

Non-game tasks (system modifications, file editing, etc.):
  [Agent ID: xxxxx]
  - Task:
  - Target:
  - Purpose:

During gameplay:
  [Rule Arbiter] Adjudicating...
  [Scenario] Checking events...
  [Worldbuilding] Registering new location...
  [Player:Noeul] Choosing action...
  [Player:Saiki] Choosing action...
  [NPC:Harlan] Generating dialogue/action...
  [NPC:Mira] Generating dialogue/action...
  [System Commit] Running gm-update + generating illustration...
  -> GM narration output

Player Agents:
- AI-controlled players (controlled_by: "ai") get one Agent [Player:{name}] each
- Example: [Player:Noeul], [Player:Saiki] (characters with controlled_by: "user" are controlled directly by the user)
- Role: observe the situation and choose actions (move, combat, dialogue, explore, etc.)
- Reads: agents/agent_{id}.json (agent identity) + entities/{scenario_id}/players/player_{id}.json (character data)
- Combines agent personality/playstyle + character background/relationships/inventory for decision-making
- Difference from NPC Agent: players strategically "choose" actions to achieve goals

Player Agent System (agents/)
===
Player agents (= AI players) and characters (= in-game personas) are separate concepts.
Agents persist, while characters can be replaced.

Structure:
  agents/                          <- Player agents (scenario-independent, persistent)
    agent_a.json                   <- Agent A's identity
    agent_b.json
    agent_c.json
  entities/{scenario}/players/     <- Characters (scenario-bound, replaceable)
    player_1.json
    player_2.json

Linking:
  agent.json's current_character -> "player_1" (currently controlled character)
  player.json's controlled_by -> "ai" (backward compatibility: "user" | "ai")
  player.json's agent_id -> "agent_a" (which agent controls this character, optional)
  * controlled_by remains "ai" for existing code compatibility
  * agent_id identifies the specific agent (bidirectional link)

Agent JSON Structure:
{
  "id": "agent_a",
  "name": "Player A",
  "gender": "male",                    // Agent's gender (may differ from character's gender)
  "personality": "...",                 // Agent's personality (decision-making style)
  "play_style": "...",                  // Play tendency (aggressive/defensive/support, etc.)
  "decision_tendency": "...",           // Decision criteria (intuitive/analytical/adventurous, etc.)
  "current_character": "player_1",      // Currently controlled character ID
  "character_history": [                // Previously controlled character records
    {"character_id": "player_1", "name": "Gaon", "status": "dead", "scenario": "lost_treasure"}
  ]
}

Character Replacement Scenarios:
1. Character death -> record in character_history -> create new character -> update current_character
2. NPC transfer -> move NPC to players/ -> change controlled_by to agent ID -> update current_character
3. Scenario transition -> link current_character to the new scenario's character

When calling Agent [Player:{name}]:
1. Read agents/agent_{id}.json -> restore agent identity
2. Read entities/{scenario}/players/player_{id}.json -> restore character state
3. Combine agent personality + character personality for decision-making
   - Agent = "What kind of player is this person?" (strategic? improvisational?)
   - Character = "What kind of person is this character?" (personality, background, motivation)
   - Combine both to determine "What would this player do as this character in this situation?"

NPC Agents:
- Each NPC gets one Agent [NPC:{name}] (e.g., [NPC:Harlan], [NPC:Mira])
- Each Agent independently determines dialogue/actions based on the NPC's personality/memory/relationships/goals
- Multiple Agents (Player + NPC) can run in parallel simultaneously

NPC Proactivity Rules:
- NPCs are not passive beings that only wait for player reactions
- NPCs act proactively based on their own goals/routines/emotions:
  - They can initiate conversations on their own
  - They can refuse player proposals
  - They can take independent actions (flee, negotiate, betray, etc.)
  - They carry on with their daily lives even when players are not around
- When calling an NPC Agent, the GM must ask "How would this NPC act in this situation?"
  (Not simply "React to this" but "Generate actions/dialogue considering this NPC's goals and circumstances")

Agent [Worldbuilding] Details
===
- Managed file: worldbuilding.json
- Auto-registration: when GM/Scenario Agent mentions new locations, NPCs, factions, items, or lore, they are automatically added to worldbuilding.json
- Conflict check: warns the GM if new content contradicts existing lore
  - Example: if "Karendel is to the north" but someone says "to the south", it is blocked
- Query support: when GM needs info on a specific region/NPC, it queries worldbuilding.json and provides the result
- Relationship management: auto-maps relational data such as distances between regions, faction relations, NPC affiliations
- Scenario-independent: world setting data persists even when scenarios change
- Invocation timing:
  - During scenario creation: register new locations/NPCs/factions
  - During GM narration: auto-detect when new lore is mentioned -> register
  - At session start: read worldbuilding.json to restore world setting context

NPC travel (Travel Information)
===
When an NPC is traveling, add a travel field. The worldbuilding agent automatically validates route plausibility.

Fields:
- origin: departure location name (location name from worldbuilding.json)
- destination: destination name
- route: planned route (array of waypoint names)
- current_segment: segment currently being traveled
- met_party_at: location where the NPC met the party
- travel_mode: mode of travel (walk/cart/horse)

Validation items:
- Whether met_party_at is included in route (met along the route)
- Whether the origin->destination route exists in connections
- Warning if detour ratio exceeds 1.8x

Agent [System Commit] Details
===
- Calls gm-update API + generates illustrations
- Updates game_state.json + entities/
- Confirms NPC registration (NPCs that appeared in narration)
- Records gm_turn.py logs
- git commit (every turn), push (every 3 turns / on explicit request)
