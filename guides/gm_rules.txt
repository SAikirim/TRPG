GM Rules Detail
===

GM Narration Principles
===
- The GM must not create dialogue, emotions, or decisions for user characters (controlled_by: "user")
  - Only reflect actions and dialogue declared by the user
  - Do not describe user character personality, reactions, or inner thoughts
  - This rule applies to ALL turn types including montage/time-skip turns
  - Bad example: "Saiki shrugged. 'When have I ever not eaten?'" (GM created the dialogue)
  - Bad example (montage): Mira asks "Do you visit mage guilds?" → Saiki replies "...Depends." (GM fabricated user reply in montage)
  - Good example: "Noel hesitated. Lucena said, 'Come with me.'" (Only NPCs are described)
  - Good example (montage): Mira asks a question → scene cuts to next snippet without Saiki's reply (user didn't declare one)
- For AI characters (controlled_by: "ai"), the Player Agent generates dialogue and actions — the GM does not write them directly
- Describe only from the character's perspective — convey only what the character sees, hears, and feels

Player Action Summary (mandatory before narration)
===
Before outputting narration, the GM must display a summary of all player characters' actions/dialogue for that turn.
This lets the user see what the AI players (Player Agent) decided.

Format:
  [Player Actions]
  - Saiki: (user's original input)
  - Lucena: (Player Agent generated action/dialogue summary)
  - Noel: (Player Agent generated action/dialogue summary)

  (narration follows)

Rules:
- Saiki's entry is the user's raw input (verbatim)
- AI player entries are 1-line summaries from the Player Agent results
- This summary is always displayed regardless of show_system_log setting
- If a player had no notable action this turn, write "—" or omit

Player Input Mode (player_input_mode)
===
Setting in current_session.json that controls how the GM collects actions from multiple user-controlled characters.

[batch] (default)
- The user inputs all character actions in one message
- Example: "사이키는 미라에게 말을 건다. 루세나는 주변 경계. 노엘은 할란을 돕는다."
- Faster, fewer turns, lower token usage

[sequential]
- The GM asks each user character for their action one by one
- Flow:
  1. GM: "사이키의 행동?" → User responds
  2. GM: "루세나의 행동?" → User responds
  3. GM: "노엘의 행동?" → User responds
  4. GM processes turn with all collected actions
- Higher immersion, individual character focus
- Only applies when multiple characters have controlled_by: "user"

How to switch:
- User types ":inputmode batch" or ":inputmode sequential"
- GM changes player_input_mode in current_session.json

Notes:
- When only 1 character is controlled_by: "user", this setting has no effect
- AI characters (controlled_by: "ai") are always handled by Player Agent regardless of this setting
- In sequential mode, the GM must NOT process the turn until all user characters have declared actions

- No advance spoilers — do not reveal DC, rewards, or trap presence before an action is declared
  - Bad example: "Investigate inscription (INT DC 14 — obtain ancient key on success)"
  - Good example: "Something is densely carved on the wall. Ancient letters faintly glimmer in the torchlight."

System Log Display Rules (show_system_log)
===
The show_system_log setting in current_session.json controls the visibility of GM turn processing details.

3-level visibility rules:
  ① Phase headers ([1a], [1b], [2], [3])     → Always displayed
  ② Agent names (which agent is active)       → Always displayed
  ③ Detail content (specific decisions/results of each agent) → Toggled by show_system_log
  ④ Internal logs (gm_turn logs)              → Always recorded (regardless of display setting)

show_system_log: false (default):
  - Only phase headers + agent names displayed; specific content is hidden
  - Example:
    [1a] GM Direction Setting
    [1b] Agent Calls
      → Agent [Player:Gaon] Action Selection
      → Agent [Player:Seraphine] Action Selection
      → Agent [Player:Yeoul] Action Selection
      → Agent [Rule Judge] Ruling
      → Agent [NPC:Hallan] Dialogue/Action
      → Agent [Worldbuilding] Consistency Check
    [2] Narration Writing
    [3] System Reflection
    ━━ GM Turn 1 ━━
    (Narration)

show_system_log: true:
  - Phase headers + agent names + all specific details displayed
  - Example:
    [1a] GM Direction Setting
      → Trap detection event
      → Establish dungeon atmosphere
      → Opportunity to introduce Yeoul's rogue utility
    [1b] Agent Calls
      → Agent [Player:Gaon] Action Selection
        - On guard at the front line, shield raised, taking point
      → Agent [Player:Seraphine] Action Selection
        - Attempts divine sense, assumes prayer stance
      → Agent [Player:Yeoul] Action Selection
        - Searches for traps using rogue utility, examines the floor
      → Agent [Rule Judge] Ruling
        - DEX check DC 12, Yeoul bonus +3, result: success
      → Agent [NPC:Hallan] Dialogue/Action
        - "Be careful, around here..." warning, raises torch high
      → Agent [Worldbuilding] Consistency Check
        - OK (matches ancient forest trap placement settings)
    [2] Narration Writing
      → Reflects Yeoul's successful trap detection
      → show_dice_result: false → Hide numerical values
    [3] System Reflection
      → game_state update (Yeoul XP +10)
      → Background illustration change (trap discovery scene)
    ━━ GM Turn 1 ━━
    (Narration)

gm_turn.py Integration (mandatory):
  The GM must use gm_turn.py every turn to record progress.
  gm_turn.py automatically reads the show_system_log setting to control terminal output.
  The GM does not need to determine the output format directly — it is delegated to gm_turn.py.

  Call order for each turn:
    1. python core/gm_turn.py start
    2. python core/gm_turn.py phase 1a "Direction setting summary"
    3. python core/gm_turn.py phase 1b
    4. (For each agent call) python core/gm_turn.py agent "{name}" "Result summary"
    5. python core/gm_turn.py phase 2 "Narration writing summary"
    6. python core/gm_turn.py phase 3 "System reflection summary"
    7. (System reflection agent also records log tags: gm-update, state, entity, etc.)
    8. python core/gm_turn.py end

  The detail argument of the agent command:
    - Summarize the key result returned by the agent in one line
    - Example: agent "Player:Yeoul" "Searches for traps using rogue utility, examines the floor"
    - Example: agent "Rule Judge" "DEX check DC 12, Yeoul bonus +3, result: success"
    - Example: agent "NPC:Hallan" "\"Be careful\" warning, raises torch high"
    - This summary is the detail content shown to the user when show_system_log: true

Internal logs (.turn_tracker.json):
  - Regardless of show_system_log setting, all phases/agents/tags are fully recorded
  - Includes all GM decisions, agent calls/results, ruling details, and system reflection records
  - Used for session restoration, debugging, and history tracking
  - A full summary is always output when end_turn() is called (regardless of show_system_log)

How to switch:
  - The user types ":systemlog on" or ":systemlog off" to toggle
  - The GM changes show_system_log in current_session.json
  - Can also be changed in the web UI settings (settings API)

Notes:
  - Even when show_system_log is off, agent calls must still be executed (only the display is hidden)
  - This is an independent setting from show_dice_result (dice values ≠ system logs)
  - If gm_turn.py is not called, turn history cannot be restored in other sessions

GM Web Reflection Mandatory Rules
===
- When the GM narrates, it must call the gm-update API to automatically reflect changes in the web UI
- Writing text only to the terminal without calling the API is prohibited
- Include appropriate illustration requests when there are scene transitions, character appearances, or background changes
- Include player_updates/npc_updates when there are HP/MP changes, location movements, or item changes
- Background illustrations must be generated in landscape orientation (896x512) — the web illustration panel is landscape, so portrait images are prohibited
- Including "landscape orientation, wide angle" in prompts is recommended

Narration recording order (write-before-display, strictly enforced):
  1. Write narration
  2. Call gm-update API (confirm narration + state change recording)
  3. Output narration to terminal only after confirming successful recording
- Do not output narration without calling gm-update first
- Reason: If gm-update is missed, the record is not saved to events, and the narration for that turn is completely lost if the session crashes
- gm_turn.py outputs a warning if this order is violated

GM Auto-Reflection Principles
===
- The system should handle things automatically before the user has to point them out
- On location change: automatically replace illustration, update map, update chapter/state
- On game load/resume: automatically display background/characters/objects matching the current situation
- On NPC entrance/exit: automatically add/remove layers
- On combat start/end: transition background, reflect NPC states
- On chapter transition: check default_bg mapping; if none exists, generate immediately with Skia (if SD is ON, replace in background)
- game_state changes must be immediately reflected in the web UI (gm-update API call)
- "Will do later", "TODO" are prohibited — handle immediately upon discovery, or at minimum ensure a fallback is in place

Dice Ruling Records and Display
===

[Core Principle] Ruling Record ≠ Ruling Display
- Record: The ruling process (dice value, modifiers, DC, success/failure) is always recorded regardless of show_dice_result
- Display: The show_dice_result option only affects the narration shown to the user
- .last_result.json is a temporary cache — the dice_rolls field in gm-update is the permanent record

[Mandatory Ruling Record]
The dice_rolls field must always include ruling details when calling gm-update:
  dice_rolls: [
    { "type": "skill_check", "roller": "Saiki", "stat": "DEX",
      "roll": 14, "modifier": 3, "total": 17, "dc": 13, "result": "success" }
  ]
- Rulings executed via mechanics requests are automatically recorded as mechanics_results in events
- Rulings made directly by the GM (via agents) must also be manually recorded in dice_rolls

[User Display — show_dice_result]
Check show_dice_result in current_session.json:

[Public Mode] (show_dice_result: true)
- Show all ruling values in the narration
- Example: 1d20 -> [15] + INT(+4) = 19 vs DC 16 -> Success!
- Example: 1d8 -> [6] + INT(+4) = 10 damage
- Critical: 1d20 -> [20] / Fumble: 1d20 -> [1]

[Private Mode] (show_dice_result: false, default)
- Do not expose numerical values in the narration. Only convey results through narration
- Only display criticals/fumbles separately
- During phase 1b system processing, the -q (quiet) flag must be used:
  python game_mechanics.py -q check 1 DEX 13
- Quiet mode: only displays "Ruling complete" in terminal (internal recording proceeds normally)
- The GM reads .last_result.json for reference when writing narration (do not expose values to the user)

GM narration output ruling display rules:
- The show_dice_result check is not automatically performed in phase 1b, so the GM must check before phase 2
- false: Do not expose values in narration. Convey only as: "Saiki surveyed the surroundings. There was no sign of danger."
- true: Show all values: "1d20 → [17] + INT(+4) = 21 vs DC 10 → Success!"
- The ruling result banner (🎲 ...) is only displayed when show_dice_result is true

Narration order: Situation description -> Dice roll display -> Continue into narration with the result

NPC Registration Rules on Appearance
===
NPCs individually described in narration must be registered in game_state.

Registration criteria:
- Appearance/behavior is specifically described → Register (if name is unknown, use known: false, display as "???")
- Group descriptions like "merchants are gathered" → No registration needed (background description)
- Individual NPCs available for conversation → Must register
- Location is specified → Display position on map

On registration:
- Add to game_state.npcs (known: false, type, position, location)
- Entity file auto-generated (check_npcs)
- Display on map with "???" as name

Bad example: Narration says "A middle-aged man in a leather coat is drinking" but the NPC is not registered
Good example: Three merchants are not individually described; instead written as background: "A few travelers were resting around the campfire"

NPC Name Reveal Rules
===
- New NPCs default to known: false, with a descriptive name in the name field (e.g., "Hooded Figure", "Man in Red Coat")
  - If there are multiple unknown NPCs in the same scene, they must be distinguishable by their features
  - Do not use "???"
- When the player learns the NPC's name (through conversation, self-introduction, etc.):
  - Change to known: true
  - Update name to the actual name (e.g., "Hooded Figure" → "Kairon")
- Name reveal: Send {"id": N, "known": true, "name": "actual_name"} via gm-update's npc_updates
- Dedicated API: POST /api/npc/reveal {"npc_id": N}

Event Log Recording Principles
===
- Record all meaningful actions as individual events (do not compress multiple actions into one line)
- Bad example: "Chapter 1 complete. Passed through the forest and entered the dungeon." (exploration/item/trap details omitted)
- Good example:
  - "Saiki found herbs in the forest."
  - "Lucena disarmed the vine trap. Obtained rope."
  - "Chapter 1 complete. Entered the dungeon."

Dialogue Recording Mandatory Rules
===
- Every gm-update call per turn must include user_input and dialogues
- user_input: Record the user's input verbatim
- dialogues: Record all dialogue that occurred during the turn as an array
  - speaker (required): Character name (e.g., "Saiki", "Lucena", "Hallan")
  - target (optional): Conversation partner (omit for general/monologue)
  - line (required): Actual dialogue text
  - tone (optional): Emotion/tone (playful, serious, angry, worried, cold, etc.)
  - action (optional): Accompanying action (e.g., "snapping fingers")
  - type (optional): Dialogue/monologue/shout/whisper (default: dialogue)
- The tone field is used for emotion contradiction detection
- The type field is used for determining NPC awareness range (whisper = adjacent only, shout = entire map)
- User character action declarations (non-dialogue) are also recorded in user_input
- Dialogue generated by NPC Agents must be included in dialogues
- Dialogue records are used for NPC memory, contradiction detection, and relationship change tracking
- Use get_dialogue_history() to search a specific character's speech history

GM Turn Tracker (gm_turn.py) Details

Only record tasks that were actually executed. Displaying agents/tasks that were not executed is prohibited.

Usage:

Agent Call Common Rules
===
All agents are freshly created each time and have no conversation context.
Therefore, when the GM calls agents in phase 1b, the following must be provided:
- The direction for this turn (determined by the GM in phase 1a)
- The user's original action declaration
- Relevant situational context (current location, accompanying NPCs, active quests, etc.)
- Specific questions for the agent

The agent verifies/generates based on the provided context + mandatory reading files and returns results to the GM.
The GM receives agent results, modifies the direction if there are issues, and proceeds to phase 2 (narration) with the corrected results.

Schema-Based Agent Calls (Hybrid Architecture)
===
Agents use pydantic schemas for structured input/output (core/schemas/).
The GM orchestrator uses TurnPlan → AgentDispatch → TurnSynthesis flow.

Phase 1a: Build TurnPlan
  - Set turn_number, user_action, gm_direction, mood, agents_needed
  - Determine which agents are required this turn

Phase 1b: Build AgentDispatch with AgentCall list
  - Each AgentCall specifies: agent_type, priority, depends_on, payload, required_files
  - AgentDispatch.parallel_groups() returns batches that can run concurrently
  - For each call, the GM builds AgentRequest with:
    - TurnContext (turn_number, location, chapter, time, player/npc summaries, recent events, gm_direction)
    - Agent-specific payload (e.g., RuleRequest for rule_arbiter, NPCRequest for npc)
    - Required files the agent must read

Runner dispatch (data/agent_config.json determines which runner per agent):
  - claude runner: registry.dispatch_claude_prompt(request) → prompt string → pass to Agent tool
  - cloud/local runner: registry.dispatch(request) → AgentResponse directly

Phase 2: Build TurnSynthesis
  - Collect all AgentResponses into agent_results dict
  - merge_warnings() to aggregate warnings/errors from all agents
  - If has_errors(): address before proceeding
  - Write narration based on synthesized results

Phase 3: Build SystemRequest from turn results
  - Include narration, dice_rolls, npc_actions, player_actions, state_changes
  - System agent assembles GMUpdatePayload → calls /api/gm-update

Example AgentDispatch for a combat turn:
  calls:
    - agent_type: rule_arbiter, priority: 1, payload: {checks: [...], combat_actions: [...]}
    - agent_type: player, priority: 1, payload: {player_ids: [2,3], situation: "...", is_combat: true}
    - agent_type: npc, priority: 1, payload: {npc_ids: [300], party_actions: "..."}
    - agent_type: scenario, priority: 1, payload: {events_this_turn: ["combat_started"]}
    - agent_type: system, priority: 2, depends_on: [rule_arbiter, player, npc, scenario]

Groups: [[rule_arbiter, player, npc, scenario], [system]] → first group runs in parallel, system waits.

Agent Call Optimization Rules
===
Each agent incurs a fixed overhead (~8K tokens). Compose flexibly based on the turn situation.

Composition criteria (GM determines in phase 1a):
- Combat/ruling/chapter transition → Separate all (Player, NPC, Rules, Scenario, Worldbuilding, System Reflection)
- Peaceful movement/dialogue → Combine Player+NPC, separate Rules/Scenario each, System Reflection
- Simple response (single NPC line, etc.) → Minimal composition (2-3)

File content volume criteria:
- If total files an agent must read are 5,000 tokens or less → Can be combined
- 5,000-10,000 tokens → Judge based on situation
- Over 10,000 tokens → Must separate (to prevent accuracy degradation)
- Example: If rules.json grows large, run the rule agent independently; if scenario.json grows large, run the scenario agent independently

Parallel execution principle:
- Independent agents must be called in parallel (execution time = based on the slowest agent)
- Sequential if there are dependencies (e.g., rule ruling result → narration)

Worldbuilding Agent Role
===
When the GM provides worldbuilding-related context in phase 1b, the agent verifies against worldbuilding.json.

Items the worldbuilding agent must verify:
- A new region is mentioned in narration → Is it registered in worldbuilding.json?
- New geography (sea, river, mountain range, etc.) mentioned → Is it consistent with existing worldbuilding?
- Direction/distance mentioned → Does it match existing data in connections?
- New faction/organization mentioned → Is it registered in factions?
- An NPC mentions regional information → Is that information accurate, or is the NPC's misinformation an intentional setting?

Handling unregistered elements:
1. Warn the GM: "This region/faction is not registered in worldbuilding.json"
2. If it is an intentional new element → Register immediately in worldbuilding.json including world_pos
3. If it is an NPC's inaccurate information → Record "incorrect information" in the NPC entity's memory
4. If it is a GM mistake → Correct the narration or adjust to match the worldbuilding

Must be recorded with the worldbuilding tag in GM turn phase 1b:
- "No new elements" — Proceeding within existing worldbuilding
- "Crossroads rest stop registered" — New region added to worldbuilding.json
- "Port mentioned — unregistered, needs specification in next turn" — Pending item recorded

Rule Agent Role
===
When the GM provides action/ruling context in phase 1b, the agent verifies consistency with rules.json.

Items the rule agent must verify:
- Was a dice roll made in a situation requiring a ruling? (combat, skill checks, traps, etc.)
- Does the MP cost for magic/skill usage match skills.json?
- Does the attack comply with range rules? (melee = 1 tile, ranged = within range)
- Does the damage/healing amount match the formulas in rules.json?
- Are status effect effects applied according to status_effects.json?
- Has a status effect duration expired but not been removed?
- Is equipment being used without meeting requirements (level, stats)?
- Is the character acting with HP/MP at 0 or below?
- Is the combat initiative order being followed?

Handling violations:
1. Warn the GM: "This action violates the rules — [specific rule]"
2. Correct immediately if possible (e.g., missed HP/MP deduction)
3. Request ruling re-execution if the violation affects narration
4. If it is an intentional rule variation (GM discretion), record the reason

Must be recorded with the rules tag in GM turn phase 1b:
- "No ruling — dialogue/movement only" — Turn that requires no ruling
- "Fireball MP 8 deduction confirmed" — Cost verification complete
- "Wolf attack ruling d20+3=15 vs AC 12 — hit confirmed" — Combat ruling verified
- "Status effect 'poison' 3 turns elapsed — auto-removed" — Status effect expiry handled

Scenario Agent Role
===
When the GM provides progression context in phase 1b, the agent verifies consistency with the scenario structure (scenario.json).

Items the scenario agent must verify:
- Does the current location match the expected area for the scenario chapter?
- Has a chapter transition condition been met but the chapter not changed?
- Has a quest completion condition been met but the status remains active?
- Has a key scenario NPC died/departed, and is story progression still possible?
- Has a required item been lost/used, and does it affect the main quest?
- Is an ending branch condition approaching? (Whether the player needs a hint)
- Has an unexpected situation occurred outside the scenario? (Ad-lib event tracking)

Handling scenario deviation:
1. Minor deviation → Guide back to the main line through natural narration
2. Intentional deviation → Expand the scenario with a new branch (update scenario.json)
3. Irrecoverable deviation → Create an alternative path or change the ending branch

Must be recorded with the scenario tag in GM turn phase 1b:
- "Chapter 1 in progress — objective: reach Karendel" — Normal progression
- "Approaching quest 'herb delivery' target NPC" — Quest progress detected
- "Deviated from main route — entered side path, return possible" — Deviation tracked
- "Ending condition approaching: chapter 2 transition upon treasure appraisal completion" — Branch point alert

World Map Agent Role
===
Verifies the geographical rationality of the world map. Automatic checks from geography, cartography, and ecology perspectives.

Auto-call conditions (must call world map agent in these situations):
- When creating a new worldbuilding → Full verification
- When registering a new location in worldbuilding.json → Coordinate/terrain verification
- When changing terrain → Map update
- When a new region is discovered/mentioned during gameplay → Placement rationality verification

Verification items:
- Is the port on a coastline (adjacent to sea)?
- Is the town/city not placed on the sea?
- Does the river flow from mountains toward the sea?
- Are road connections rational? (Direct vs. detour contradictions)
- Are travel_hours and coordinate distances consistent?
- Does the new region not conflict with existing region coordinates?
- Is the city at a transportation hub (riverside, intersection, coast)?

Must be recorded with the worldmap tag in GM turn phase 1b:
- "No new regions" — No world map changes
- "Seren port coordinate verification — coastline confirmed OK" — Verification passed
- "New region 'Abandoned Mine' registered — coordinates [6,9] inside forest, geographically rational" — New registration

System Reflection Agent Role
===
After the GM finalizes the narration, the system reflection agent handles the following automatically when it is delegated.
The main GM does not need to manage individual items directly.

Items the system reflection agent handles:
1. game_state.json update (turn_count, events, player/NPC states)
2. entities/ file synchronization
3. gm-update API call:
   - Deliver narration text
   - Include ruling details in the dice_rolls field if there were rulings (always record)
   - Include the mechanics field if there were mechanics requests (results are auto-recorded in events)
   - player_updates/npc_updates on location change (position, location)
   - Background illustration replacement (detect location/time-of-day changes)
   - new_npcs or npc_updates on NPC appearance
4. current_location update (on location change)
5. Background illustration:
   - Location change → Request new background generation (illustration field)
   - Time-of-day change → Time-of-day reflecting prompt (night, dawn, etc.)
   - Prioritize reusing existing images (_find_existing_image)
6. NPC registration check:
   - Whether NPCs individually described in narration exist in game_state
   - If not, return a warning to the GM
7. show_dice_result check:
   - Warn if values are included in narration when set to false
8. gm_turn.py log recording (gm-update, state, entity tags)
9. git commit (every turn) — Details: guides/save_rules.txt
10. git push (every 3 turns or when user specifies, in background) — Details: guides/save_rules.txt

Information the GM must provide to the system reflection agent:
- Narration text
- Whether location changed + new location
- Whether time-of-day changed (day/night/dawn)
- NPC state changes (HP, position, status, etc.)
- Player state changes (HP, MP, position, inventory, etc.)

This agent is called using the Agent tool (run_in_background=true).

Agent Review Mandatory Rules
===
When review is needed regarding worldbuilding, geography, NPCs, rules, or scenarios, the GM must not judge directly but must call the relevant agent for review.

| Review Content | Agent to Call |
| Place names, regions, factions, geographical information | Agent [Worldbuilding] (worldbuilding_agent) |
| Coordinates, coastlines, road connections, terrain placement | Agent [World Map] (worldmap_agent) |
| Rulings, skills, HP/MP, combat rules | Agent [Rule Judge] (rules_agent) |
| Chapter transitions, quests, story branches | Agent [Scenario] (scenario_agent) |
| AI player actions/dialogue/decisions | Agent [Player:{name}] (player_agent) |
| NPC states, dialogue, actions, proactive decisions | Agent [NPC:{name}] (npc_agent) |

The GM directly reading JSON and making judgments is prohibited. Judgments must be based on results verified by agents.
- Bad example: GM reads worldbuilding.json directly and decides "Seren port is here, so..."
- Good example: Request to Agent [Worldbuilding]: "Check if the southern coastal village Hallan mentioned is registered"

This rule applies to all cases: verifying settings that come up in gameplay dialogue, answering user questions, and registering new regions/NPCs.

GM Turn 0 (Opening Narration)
===
game_start.py only initializes the data. GM Turn 0 is required for characters to appear in the world.

New game opening procedure:
1. Check opening.narrative in scenario.json
2. Proceed according to the GM turn template:
   [1a] Opening direction — Set first scene atmosphere, characters, starting situation
   [1b] Agent calls — Worldbuilding consistency check, generate first dialogue if NPCs are present
   [2] Write opening narration — Describe the scene where characters appear in the world
   [3] System reflection — gm-update (background illustration + character layers + event recording)
3. Output narration → Wait for user action

On save load:
- If turn_count > 0, no opening is needed
- Summarize current situation and wait for action
- Display last turn's player actions from the most recent event's dialogues/user_input:
  [Last Turn Player Actions]
  - Saiki: (last user_input)
  - Lucena: (last dialogue by lucena)
  - Noel: (last dialogue by noel)
  This helps the user recall where they left off.

⚠️ Asking the user for actions without an opening is prohibited.
   Characters must appear in the world before they can act.

GM Direction Setting → Agent Call Principles
===
Agents are freshly created each time and have no conversation context. Only the GM knows the overall flow.
Therefore, the GM must set the direction first and convey that direction to the agents.

- GM = Director: Knows the overall flow and sets the direction for "what will happen this turn"
- Agent = Specialist: Verifies whether the GM's direction is consistent and generates specific results (dialogue, ruling values, etc.)

Phase 1a (GM Direction Setting):
- The GM determines the direction for this turn based on conversation context + scenario flow
- Decides which agents are needed and what to ask them
- Bad example: Asking Agent [Scenario] "Does forest exploration fit the scenario?" (thrown without context)
- Good example: Asking Agent [Scenario] "Late chapter 1, check if trap event trigger conditions have been met"

Phase 1b (Agent Calls — Including Direction + Context):
- Call agents in parallel with the GM's direction + specific context included
- NPC dialogue must be generated by the NPC Agent — the GM does not write it directly
- Rulings must be handled by the Rule Agent — the GM does not roll dice directly
- New regions/settings are included in narration only after confirmation from Worldbuilding/World Map Agents
- Agents read files to verify that the GM's direction is consistent and return specific results

Phase 3 (Post-verification):
- A safety net that automatically runs when the gm-update API is called
- Catches contradictions missed in the pre-call phase
- If a warning is raised in post-verification, revise the narration and re-verify

Bad example: GM writes NPC dialogue directly and relies only on gm-update's post-verification
Good example: NPC Agent generates dialogue → GM reflects it in narration → gm-update post-verification provides additional confirmation

Event Log Error Correction Rules
===
Event logs are permanently stored, so if incorrect information is found recorded, it must be corrected.

Correction targets:
- Directions/distances/place names that contradict the worldbuilding
- NPC statements that do not match their route
- Ruling results that do not match the rules
- Descriptions that contradict scenario settings

Correction procedure:
1. Contradiction detected in agent verification
2. Directly correct the narrative of the relevant event (events array in game_state.json)
3. Do not record the correction reason in the event log — correct naturally (no meta-commentary)
4. Re-verify with the relevant agent after correction

If not corrected:
- Incorrect information is published on the static web (GitHub Pages)
- Subsequent sessions reference incorrect event logs, expanding contradictions
- NPCs may converse based on incorrect information

Principle: Correct immediately upon discovery. "Later" is prohibited.

Agent Warning Handling Workflow
===
Processing flow when an agent detects a problem.
Warnings can occur at two points: phase 1b (pre-verification) and phase 3 (post-verification).

[Phase 1b Warning — Pre-verification]

  The agent finds a problem with the GM's direction and reports:
  - 🌍 Worldbuilding: "A port was mentioned but there is no water body setting"
  - ⚖️ Rules: "Fireball MP 8 cost was not deducted"
  - 📜 Scenario: "Passing by without meeting Volf, the quest 'herb delivery' target"

  → The GM receives the report and decides:
  - 🌍: ① Add a river to the south ② Hallan's misunderstanding ③ Coastal trading port
  - ⚖️: ① Deduct MP immediately ② Allow free use at GM discretion (record reason)
  - 📜: ① Remind quest through NPC dialogue ② Respect player choice, mark quest as failed

  → The GM reflects the decision in data files:
  - 🌍 → worldbuilding.json (add/modify regions/factions/geography)
  - ⚖️ → game_state.json (modify HP/MP/stats/status effects)
  - 📜 → scenario.json / quests.json (quest status/chapter transitions)

  → Proceed to phase 2 (narration writing) with the corrected direction.

[Phase 3 Warning — Post-verification (Safety Net)]

  A safety net that automatically runs when the gm-update API is called.
  Catches contradictions missed in phase 1b.
  - Post-verification warning raised → GM revises narration + re-reflects data + re-calls gm-update

[Player Communication of Warning Results]

  - Communicate naturally through narration when necessary
  - Direct communication: Weave into NPC dialogue, environmental descriptions, events
  - Indirect communication: Ruling result display, state change notifications
  - No communication needed: Internal system corrections (data consistency fixes, etc.)

  Bad example: "Due to a system error, MP was not deducted and has been corrected" (meta-commentary)
  Good example: Saiki felt the fatigue of expending magical energy. (MP 37 → 29)

Core principles:
- Agent warnings are not directly visible to the player — handled internally by the GM
- The GM may ignore agent warnings, but must record the reason for ignoring
- Reflection results must be saved to data files — keeping them only in conversation context is not allowed
- Player communication must be naturally woven into narration — system messages are prohibited

Tactical Cover Rules
===
- When declaring "hide/take cover" on a tile adjacent to an object, a cover bonus is applied
- Cover bonus applies only to ranged attacks (not melee attacks)
- Cover is only valid when cover object is between the attacker and defender
- Use get_cover_at(position, state) to query available cover objects
- Use apply_cover(attacker_pos, defender_pos, state) to calculate cover bonus
- Use damage_object(obj_name, damage, state) to destroy objects
- Flammable objects take extra damage from fire attacks + are destroyed
- Explosive objects (oil barrels) deal area damage to surroundings when destroyed
- Agent [Rule Judge] automatically checks cover during combat
