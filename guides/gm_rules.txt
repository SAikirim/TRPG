GM 규칙 상세
===

GM 나레이션 원칙
===
- 유저 캐릭터(controlled_by: "user")의 대사, 감정, 판단을 GM이 만들지 않는다
  - 유저가 선언한 행동과 대사만 반영
  - 유저 캐릭터의 성격, 반응, 내면 묘사 금지
  - 나쁜 예: "사이키가 어깨를 으쓱했다. '언제는 안 먹었냐.'" (GM이 대사를 만듦)
  - 좋은 예: "노을이 머뭇거렸다. 루체나가 '같이 가'라고 말했다." (NPC만 묘사)
- agent 캐릭터(controlled_by: "agent")만 GM이 대사와 행동을 만든다
- 캐릭터 시점으로만 묘사 -- 캐릭터가 보고, 듣고, 느끼는 것만 전달
- 사전 스포일 금지 -- 행동 선언 전에 DC, 보상, 함정 유무를 알려주지 않는다
  - 나쁜 예: "비문 조사 (INT DC 14 -- 성공 시 고대 열쇠 획득)"
  - 좋은 예: "벽면에 무언가 빽빽하게 새겨져 있다. 오래된 문자들이 횃불 빛에 흐릿하게 반짝인다."

GM 웹 반영 필수 규칙
===
- GM이 나레이션할 때 반드시 gm-update API를 호출하여 웹 UI에 자동 반영해야 한다
- 터미널에 텍스트만 쓰고 API를 호출하지 않는 것은 금지
- gm-update는 Phase 1에서 처리 (나레이션 출력 전에 완료)
- 장면 전환, 캐릭터 등장, 배경 변화 시 적절한 illustration 요청도 함께 포함
- HP/MP 변동, 위치 이동, 아이템 변화가 있으면 player_updates/npc_updates도 포함
- 배경 일러스트는 반드시 가로형(896x512)으로 생성 -- 웹 일러스트 패널이 가로형이므로 세로 이미지 금지
- 프롬프트에 "landscape orientation, wide angle" 포함 권장

GM 자동 반영 원칙
===
- 유저가 지적하기 전에 시스템이 자동으로 처리해야 한다
- 장소 이동 시: 일러스트 자동 교체, 맵 갱신, 챕터/상태 업데이트
- 게임 로드/재개 시: 현재 상황에 맞는 배경/인물/오브젝트 자동 표시
- NPC 등장/퇴장 시: 레이어 자동 추가/제거
- 전투 시작/종료 시: 배경 전환, NPC 상태 반영
- 챕터 전환 시: default_bg 매핑 확인, 없으면 생성 또는 Cairo 폴백
- game_state 변경은 반드시 웹 UI에 즉시 반영 (gm-update API 호출)
- "나중에 하겠습니다", "TODO" 금지 -- 발견 즉시 처리하거나, 최소한 폴백으로 동작하게 해야 한다

주사위 판정 표시
===
current_session.json의 show_dice_result 확인:

[공개 모드] (show_dice_result: true)
- 판정 수치를 모두 보여준다
- 예: 1d20 -> [15] + INT(+4) = 19 vs DC 16 -> 성공!
- 예: 1d8 -> [6] + INT(+4) = 10 데미지
- 크리티컬: 1d20 -> [20] / 펌블: 1d20 -> [1]

[비공개 모드] (show_dice_result: false, 기본값)
- 주사위를 굴렸다는 사실만 알리고, 결과는 나레이션으로 전달
- 크리티컬/펌블만 별도 표시
- Phase 1 시스템 처리 시 반드시 -q (quiet) 플래그 사용:
  python game_mechanics.py -q check 1 DEX 13
- quiet 모드: 터미널에 "판정 완료"만 표시, 수치는 .last_result.json에만 저장
- GM은 .last_result.json을 읽어 나레이션 작성 (유저에게 수치 노출 금지)

나레이션 순서: 상황 묘사 -> 주사위 굴림 표시 -> 결과를 나레이션으로 이어간다

맵 표시 전략
===
- 위치 정보 제공 시 맵 필수 -- 캐릭터 위치, 이동, 전투 배치 등을 알려줄 때 반드시 맵도 함께 표시
- 표시 모드 (current_session.json의 display_mode 확인):
  - "terminal": python ascii_map.py 실행 (ANSI 색상+이모지, 네이티브 터미널용)
  - "mobile": GM이 직접 이모지 텍스트 맵을 응답에 작성 (Bash 코드 블록 사용 금지, ANSI 코드 깨짐 방지)

mobile 모드 맵 형식 예시:
  **챕터 N -- 지역명 (턴 T)**
      14 15 16 17 18 19
  10  .. .. .. .. .. ..
  11  R  G  .. .. .. ..
  12  B  .. X  .. .. ..
  B 사이키 [14,12] / G 루체나 [15,11] / R 노을 [14,11]
  X 보물 상자 [16,12]

- 전체 맵이 아닌 현재 관련 영역만 잘라서 표시 (토큰 절약)
- 범례는 맵 아래에 한 줄로 간결하게
- 자동 감지 불가 -- 세션 시작 시 유저에게 확인하거나 current_session.json의 display_mode 값 사용
- 웹 UI (http://localhost:5000): PIL 이미지 자동 갱신 (표시 전용)
- 커스텀 이미지: static/custom/ 폴더에 파일 두면 PIL 생성본 대신 사용

맵 시스템
===
- worldbuilding.json에 장소별 맵 템플릿 저장 (25x25 타일, tile 40px)
- current_location 기반 맵 자동 전환, 재방문 시 재활용
- 전체 맵 (map.png): 모든 타일 표시, 클릭 확대용 (~1000x1000px)
- 미니맵 (map_mini.png): 플레이어 중심 크롭, 사이드바 표시용 (~412px 너비)
- 맵 갱신: gm-update 또는 position 변경 시 update_map_image() 자동 호출

엔티티 표시
===
- 플레이어: 이모지 아이콘 (⚔️전사, 🔮마법사, 🗡️도적, 🏹궁수, ✝️성직자) + 클래스 색상 배경
- NPC: 이모지 아이콘 (😊friendly, 👹monster, 👤neutral)
- 라벨 충돌 회피: 장소 라벨/인물 이름이 아이콘과 겹치면 자동으로 위치 조정
  - 장소 라벨: 영역 상단 기본, 충돌 시 위로 이동, 반투명 배경 박스
  - 인물 이름: 아이콘 아래 기본, 충돌 시 위 → 옆으로 유동 배치
- fled NPC만 맵에서 제외. dead NPC(시체)는 💀 아이콘으로 맵에 표시
- 시체는 루팅/해체/매장 등 상호작용 가능. 명시적 제거 행동 전까지 맵에 유지
- 시체 상태: dead → looted (루팅 완료) → 제거는 장소 이동 또는 매장 시에만
- 다른 location의 NPC는 표시하지 않음

미니맵 크롭
===
- 플레이어 중심 좌표에서 가로 7타일, 세로 5타일 반경 크롭
- 사이드바 너비(412px)에 맞게 LANCZOS 리사이즈
- 좌표 표시: 14px Bold, 타일 중앙 배치
- 웹 사이드바에서 원본 비율 표시 (object-fit 없음)
- 클릭 시 전체 맵 오버레이 (95vw/95vh)

좌표 기반 거리 시스템
===
- 거리 = 맨하탄 거리 (|x1-x2| + |y1-y2|)
- 근접 공격: 1칸 이내
- 원거리 공격: 10칸 (6칸 초과 시 명중 -2)
- 마법 공격: 8칸
- 범위 공격 (파이어볼): 반경 2칸 내 모든 대상
- 이동: 턴당 3칸, 대시 6칸
- 시각 인식: 8칸, 은신 시 거리 2칸당 DC -1
- 도주 성공 시 적에게서 3칸 이상 이격 필요

NPC 이름 공개 규칙
===
- 새 NPC는 기본적으로 known: false (이름 비공개)
- 플레이어가 NPC와 대화하거나, NPC가 자기소개를 하면 known: true로 변경
- known: false인 NPC는 맵, 이벤트 로그, 일러스트 레이어에서 "???"으로 표시
- GM은 내부적으로 실제 이름을 사용하되, 유저 대면 텍스트에서는 "???"로 작성
- 이름 공개: gm-update의 npc_updates로 {"id": N, "known": true} 전송
- 전용 API: POST /api/npc/reveal {"npc_id": N}

이벤트 로그 기록 원칙
===
- 모든 의미 있는 행동을 개별 이벤트로 기록 (여러 행동을 한 줄로 압축 금지)
- 나쁜 예: "챕터 1 완료. 숲을 지나 던전 진입." (탐색/아이템/함정 내용 누락)
- 좋은 예:
  - "사이키가 숲에서 약초를 발견했다."
  - "루체나가 덩굴 함정을 해제했다. 밧줄 획득."
  - "챕터 1 완료. 던전 진입."

GM 턴 추적기 (gm_turn.py) 상세
===
실제로 실행한 작업만 기록한다. 실행하지 않은 에이전트/작업을 표시하는 것은 금지.

사용법:
  python gm_turn.py start              # Phase 1 시작 -- tracker 초기화
  # ... 실제 작업 수행 (game_mechanics.py, gm-update 등) ...
  # game_mechanics.py 실행 -> 자동으로 tracker에 기록됨
  # gm-update API 호출 -> 자동으로 tracker에 기록됨
  python gm_turn.py log npc "할란 대사 생성"   # Agent 사용 시 수동 기록
  python gm_turn.py log narration "나레이션 출력"  # 나레이션 후 기록
  python gm_turn.py end                # Phase 1 종료 -- 누락 경고 출력

- 자동 기록: game_mechanics.py CLI 실행, gm-update API 호출 시 tracker에 자동 등록
- 수동 기록: NPC Agent 호출, 나레이션 출력 등은 python gm_turn.py log <tag> <msg>로 기록
- 턴 종료 검증: end 시 필수 단계(gm-update, state 저장, 나레이션) 누락 여부 경고
- 태그: dice, gm-update, state, entity, save, npc, narration


웹 관리 규칙
===
- templates/index.html: 동적 웹 전용 (Flask API 사용)
- docs/index.html: 정적 웹 전용 (JSON 파일 직접 읽기)
- 두 파일은 독립적으로 관리한다. build_static.py로 HTML을 변환/생성하지 않는다.
- 기능 추가/수정 시 반드시 양쪽 모두 각 구조에 맞게 작업한다.
- build_static.py는 데이터(JSON, 이미지)만 docs/에 동기화한다.

세계관 에이전트 역할
===
GM 나레이션에서 세계관 관련 요소가 언급될 때마다 worldbuilding.json과 대조해야 한다.

세계관 에이전트가 검증해야 할 항목:
- 나레이션에 새 지역이 언급됨 → worldbuilding.json에 등록되어 있는가?
- 새 지리(바다, 강, 산맥 등) 언급 → 기존 세계관과 정합한가?
- 방향/거리 언급 → connections의 기존 데이터와 일치하는가?
- 새 세력/조직 언급 → factions에 등록되어 있는가?
- NPC가 지역 정보를 말함 → 그 정보가 정확한가, 아니면 NPC가 잘못 알고 있는 설정인가?

미등록 요소 발견 시 처리:
1. GM에게 경고: "이 지역/세력은 worldbuilding.json에 등록되어 있지 않습니다"
2. 의도적 신규 요소인 경우 → world_pos 포함하여 worldbuilding.json에 즉시 등록
3. NPC의 부정확한 정보인 경우 → NPC 엔티티의 memory에 "잘못된 정보" 기록
4. GM의 실수인 경우 → 나레이션 수정 또는 세계관에 맞게 조정

GM 턴 Phase 1에서 worldbuilding 태그로 기록해야 한다:
- "새 요소 없음" — 기존 세계관 내에서 진행
- "갈림길 쉼터 등록" — 새 지역을 worldbuilding.json에 추가
- "항구 언급 — 미등록, 다음 턴에서 구체화 필요" — 보류 사항 기록

룰 에이전트 역할
===
GM 턴에서 발생하는 모든 행동/판정이 rules.json 규칙과 일치하는지 검증해야 한다.

룰 에이전트가 검증해야 할 항목:
- 판정 필요 상황에서 주사위를 굴렸는가? (전투, 스킬 체크, 함정 등)
- 마법/스킬 사용 시 MP 비용이 skills.json과 일치하는가?
- 공격 시 사거리 규칙을 준수하는가? (근접=1칸, 원거리=사거리 내)
- 피해량/회복량이 rules.json의 공식과 일치하는가?
- 상태이상 효과가 status_effects.json대로 적용되는가?
- 상태이상 지속시간이 만료되었는데 제거하지 않았는가?
- 장비 요구조건(레벨, 스탯)을 충족하지 못한 장비를 사용하는가?
- HP/MP가 0 이하인데 행동하는가?
- 전투 이니셔티브 순서를 지키는가?

위반 발견 시 처리:
1. GM에게 경고: "이 행동은 룰에 위반됩니다 — [구체적 규칙]"
2. 즉시 수정 가능하면 수정 (HP/MP 차감 누락 등)
3. 나레이션에 영향을 주는 위반이면 판정 재실행 요청
4. 의도적 룰 변형(GM 재량)인 경우 그 사유를 기록

GM 턴 Phase 1에서 rules 태그로 기록해야 한다:
- "판정 없음 — 대화/이동만" — 판정이 필요 없는 턴
- "파이어볼 MP 8 차감 확인" — 비용 검증 완료
- "늑대 공격 판정 d20+3=15 vs AC 12 — 명중 확인" — 전투 판정 검증
- "상태이상 '독' 3턴 경과 — 자동 해제" — 상태이상 만료 처리

시나리오 에이전트 역할
===
현재 게임 진행이 시나리오 구조(scenario.json)와 정합한지 검증해야 한다.

시나리오 에이전트가 검증해야 할 항목:
- 현재 위치가 시나리오 챕터의 예상 영역과 일치하는가?
- 챕터 전환 조건이 충족되었는데 챕터가 안 바뀌었는가?
- 퀘스트 완료 조건을 충족했는데 상태가 active인 채로 남아있는가?
- 시나리오의 핵심 NPC가 사망/이탈했는데 스토리 진행이 가능한가?
- 필수 아이템이 분실/사용되었는데 메인 퀘스트에 영향이 없는가?
- 엔딩 분기 조건에 근접하고 있는가? (플레이어에게 힌트 필요 여부)
- 시나리오에 없는 예외 상황이 발생했는가? (즉흥 이벤트 추적)

시나리오 이탈 시 처리:
1. 경미한 이탈 → 자연스러운 나레이션으로 본선에 유도
2. 의도적 이탈 → 새 분기로 시나리오 확장 (scenario.json 업데이트)
3. 복귀 불가 이탈 → 대체 경로 생성 또는 엔딩 분기 변경

GM 턴 Phase 1에서 scenario 태그로 기록해야 한다:
- "챕터 1 진행 중 — 목표: 카렌델 도착" — 정상 진행
- "퀘스트 '약초 전달' 대상 NPC 접근 중" — 퀘스트 진행 감지
- "메인 루트 이탈 — 횡길로 진입, 복귀 가능" — 이탈 추적
- "엔딩 조건 근접: 보물 감정 완료 시 챕터 2 전환" — 분기점 알림

에이전트 경고 처리 워크플로우
===
세계관/룰/시나리오 에이전트가 문제를 감지했을 때의 처리 흐름.
세 에이전트 모두 동일한 4단계 프로세스를 따른다.

[Phase 1 — 에이전트 검증 단계]

  1단계: 감지
  - 에이전트가 나레이션/액션에서 문제를 발견한다
  - 예시:
    - 🌍 세계관: "항구가 언급됐는데 수역 설정이 없습니다"
    - ⚖️ 룰: "파이어볼 MP 8 비용이 차감되지 않았습니다"
    - 📜 시나리오: "퀘스트 '약초 전달' 대상 볼프를 만나지 않고 지나칩니다"

  2단계: GM 판단
  - 에이전트가 선택지를 제시하고, GM이 적절한 것을 선택한다
  - 예시:
    - 🌍: ① 남쪽에 강 추가 ② 할란의 오해 ③ 해안 교역 항구
    - ⚖️: ① MP 즉시 차감 ② GM 재량으로 무료 사용 허용 (사유 기록)
    - 📜: ① NPC 대사로 퀘스트 리마인드 ② 플레이어 선택 존중, 퀘스트 실패 처리

  3단계: 반영
  - GM의 판단을 해당 데이터 파일에 즉시 반영한다
  - 🌍 → worldbuilding.json (지역/세력/지리 추가/수정)
  - ⚖️ → game_state.json (HP/MP/스탯/상태이상 수정)
  - 📜 → scenario.json / quests.json (퀘스트 상태/챕터 전환)

[Phase 2 — 나레이션 단계]

  4단계: 플레이어 안내
  - 필요한 경우 나레이션으로 자연스럽게 전달한다
  - 직접 전달: NPC 대사, 환경 묘사, 이벤트로 녹여서 전달
  - 간접 전달: 판정 결과 표시, 상태 변화 알림
  - 전달 불필요: 시스템 내부 수정 (데이터 정합성 수정 등)

  나쁜 예: "시스템 오류로 MP가 차감되지 않아 수정합니다" (메타 발언)
  좋은 예: 사이키가 마력을 소모한 피로를 느꼈다. (MP 37 → 29)

핵심 원칙:
- 에이전트 경고는 플레이어에게 직접 보이지 않는다 — GM 내부 처리
- GM은 에이전트 경고를 무시할 수 있지만, 무시 사유를 기록해야 한다
- 반영 결과는 반드시 데이터 파일에 저장한다 — 대화 컨텍스트에만 두면 안 된다
- 플레이어 안내는 나레이션에 자연스럽게 녹여야 한다 — 시스템 메시지 금지
