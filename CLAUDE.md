# TRPG System (Claude GM)

A TRPG system where Claude serves as the GM, running through the Claude Code CLI terminal.
The web UI is **display-only** (Flask + PIL). Actual gameplay happens exclusively in the terminal.
By swapping scenarios and rulesets, you can play in various settings including fantasy, modern, horror (CoC), and more.

---

## ⚠️ Core Rules Summary (Required Reading for Every Session)

> **This section must be read and followed at the start of every session.**

### Strictly Prohibited
- ❌ The GM must not create dialogue/emotions/decisions for user characters (controlled_by: "user")
- ❌ Writing narration only in the terminal without reflecting it on the web (gm-update)
- ❌ "I'll do it later", "TODO" — handle immediately upon discovery or use fallback
- ❌ Mixing narration and system output in the same block (phase separation mandatory, narration must use bold separation)
- ❌ Using opaque backgrounds on character layers over the background
- ❌ The same character's appearance changing between scenes
- ❌ The GM making direct decisions without going through agents (for worldbuilding/rules/NPC matters)
- ❌ Exposing check numbers in narration when show_dice_result: false (internal recording is always mandatory regardless of display setting)
- ❌ Performing a check but not recording it in events (recording a check ≠ displaying a check)
- ❌ Overwriting or deleting save files without backup
- ❌ Saving to a slot that already has a save without confirmation (must use an empty slot or active_save_slot)
- ❌ Mixing data from different scenarios in a save (scenario_id mismatch)
- ❌ Manipulating save data by guessing (use actual play data)

### Required Automation (Handled by Code)
- ✅ On Flask start/load: `restore_scene()` → automatic background restoration
- ✅ Image generation: SD WebUI first → Skia fallback (no blank screens), image reuse (prioritize existing high-quality), automatic portrait background removal
- ✅ Auto-sync docs/ on save (GitHub Pages) — build_static.py copies HTML + syncs data (JSON, images)
- ✅ Auto-reject overwrite on inactive slots (active slot allowed, inactive blocked + empty slot suggestion)
- ✅ Auto-backup before save overwrite (`.backups/`, max 5)
- ✅ Auto-verify scenario_id consistency on save/load
- ✅ session_validator verifies all save file consistency (scenario_id, NPC contamination, location)
- ✅ Auto-generate NPC entities (check_npcs) — safety net for NPCs the GM mentioned but forgot to register
- ✅ Auto-carry-over world NPC states across linked scenarios (death/memory/relationships)
- ✅ On new game: only clear monster entities, preserve world NPCs (friendly/neutral)
- ✅ session_validator verifies NPC temporal consistency (previous scenario dead NPCs ↔ current state)
- ✅ Auto-switch map by location (current_location → worldbuilding.json)

### Required Every Turn (GM Must Always Follow)
- ❗ AI player actions/dialogue are generated by the Player Agent — the GM does not write them directly
- ❗ NPC dialogue/actions are generated by the NPC Agent — the GM does not write them directly (NPCs act proactively)
- ❗ Checks are handled by the Rule Agent — the GM does not roll dice directly
- ❗ Check details (dice values, DC, modifiers, results) are always recorded in the dice_rolls field of gm-update
- ❗ Background illustrations must be updated when location changes (also reflect time of day: day/night/dawn)
- ❗ World map data changes require review by the worldbuilding/world map agent (details: guides/illustration.txt 'World Map Final Review')
- ❗ NPCs individually described in narration must be registered in game_state (auto-generate is a safety net — GM is primarily responsible for registration; or handled as background description)
- ❗ New settings (place names/NPCs/routes) must be confirmed with agents before including in narration
- ❗ If incorrect information is found in event logs, correct it immediately

## GM Turn Processing Order (Strict Compliance)

> GM = director (knows the overall flow and sets direction), agents = specialists (verify direction consistency + generate specific results)
> Agents are freshly created each time and have no conversation context. Only the GM knows the overall flow.
> Therefore, the GM must set direction first, then communicate that direction to agents.

```
User declares action
 → [1a] GM sets direction (event/mood/development decision, agent selection)
 → [1b] Parallel agent calls (send direction + context + questions → receive verification + results)
 → [2]  Write narration (synthesize agent results)
 → [3]  System reflection (Agent [System Reflection]: state/gm-update/illustrations/logs/git)
 → Output narration + display map + await action
```

> Details (explanation + examples for each phase): see guides/gm_rules.txt

### GM Turn Template (Follow This Order Every Turn)

> **gm_turn.py is mandatory**: Track each turn with `start → phase/agent/log → end` calls.
> gm_turn.py automatically controls terminal output based on the show_system_log setting.
> Core flow: `gm_turn.py start` → phase 1a (direction) → phase 1b (agents) → phase 2 (narration) → phase 3 (system reflection) → `gm_turn.py end`
> show_system_log: phase headers/agent names are always shown, detailed content is on/off, internal logs are always recorded.
> Detailed template + show_system_log output examples: see guides/gm_rules.txt

## Agent Division Structure

```
User (declares actions via terminal chat)
  ↓
Main Claude = GM Orchestrator (narration/progression/user interaction)
  ├── Agent [Rule Arbiter]    — checks, dice, combat rule verification
  ├── Agent [Scenario]        — chapter/event/ending branches, quest status
  ├── Agent [Worldbuilding]   — place names/currency/factions/NPCs, consistency verification
  ├── Agent [Player:{name}]   — AI player decision-making/dialogue/actions (excludes user)
  ├── Agent [NPC:{name}]      — NPC dialogue/actions/proactive decisions
  ├── Agent [World Map]       — coordinate/geography verification, routes, map updates
  └── Agent [System Reflection] — game_state, gm-update, illustrations, logs, git
  ↓
Synthesize results → narration + map output + game_state.json update + save (git push)
```

> Required reading files, detailed roles, and personas for each agent: see guides/entities.txt

> **Agent Required Reading Rule**: All sub-agents must read their "required reading" files first before operating when called. Operating without reading the guides is prohibited.

> **Agent Call Common Rule**: Agents are freshly created each time and have no conversation context. The GM must send direction + context + specific questions in phase 1b. Details: see guides/gm_rules.txt.

### Hybrid Multi-Agent Architecture (Runner Abstraction)

Sub-agents can run on different backends. The GM sends the same pydantic-schema request regardless of backend.

```
AgentRequest (pydantic)  →  AgentRegistry (dispatcher)  →  AgentResponse (pydantic)
                                    │
                          ┌─────────┼─────────┐
                     ClaudeRunner  CloudRunner  LocalRunner
                     (Agent tool)  (Gemini/    (Ollama/
                                   Groq/       llama.cpp)
                                   OpenAI)
```

**Config**: `data/agent_config.json` — per-agent runner/model assignment.

| Runner | When to Use | Requirements |
|--------|-------------|--------------|
| `claude` (default) | No external API needed. Uses Claude Code Agent tool | None |
| `cloud` | Offload to external LLMs (Gemini Flash, Llama3, GPT) | API key in env |
| `local` | Local models via Ollama | Ollama running |

**Schema files**: `core/schemas/` — Request/Response types per agent.
**Runner files**: `core/runners/` — Backend implementations + registry.

> The GM operates as an **Orchestrator**: set direction (TurnPlan) → dispatch agents (AgentDispatch) → synthesize results (TurnSynthesis) → narrate.
> When using `claude` runner: GM builds prompt via `registry.dispatch_claude_prompt(request)` and passes it to Agent tool.
> When using `cloud`/`local` runner: GM calls `registry.dispatch(request)` for programmatic execution.

## Save Management Rules

> **Saves = the user's play records. Protect them with multiple layers of safety.**

### Quadruple Safety Net
1. **Overwrite prevention** (code): Auto-reject if an inactive slot already has a save + suggest empty slot. Active slot (active_save_slot) is auto-allowed
2. **Auto-backup** (code): save_manager auto-backs up to `.backups/` before overwrite (max 5)
3. **Consistency verification** (code): Auto-check scenario_id match on save/load, reject on mismatch
4. **Session verification** (manual): `session_validator.py` verifies scenario_id, NPC, and location across all save files

### Mandatory Principles When Modifying Save Data
- ❗ **Use actual play data** — do not fabricate by guessing
- ❗ **Always back up before modifying** — code does this automatically, but also mandatory for manual operations
- ❗ **scenario_id must match the folder name** — no karendel_journey data in `saves/lost_treasure/`
- ❗ **NPCs from other scenarios must not be mixed in** — scenario-specific NPCs belong only in that scenario's save
- ❗ **When separating scenarios, separate saves too** — maintain independent saves per scenario
- ❗ **Starting a new game does not delete existing saves** — existing saves are always preserved
- ❗ **Maintain NPC temporal continuity across linked scenarios** — NPCs dead in a previous scenario remain dead in the next

### NPC Temporal Continuity (Linked Scenarios)
- **Continue**: `_carry_over_npc_states()` auto-carries over dead/memory/relationships (excludes monsters)
- **New game**: NPC initial state (time reset). session_validator [11] auto-detects temporal contradictions

### Save Structure
```
saves/
  {scenario_id}/
    progress.json           - Progress history (turn/chapter/description records)
    slot_1/
      save.json             - Save data (save_info + game_state)
      .backups/             - Auto-backups (stored before overwrite, max 5)
        save.json.20260321_035021.bak
    slot_2/ ...
```

## Session Procedures

### Common Steps (Both New Game and Load)
1. Read CLAUDE.md (this file) — especially "Core Rules Summary" + "Required Every Turn" + "GM Turn Template"
2. Run `python session_validator.py` — state verification + auto-fix
3. Read `data/current_session.json` → `data/worldbuilding.json` → `data/game_state.json`
4. Read `guides/gm_rules.txt` — verify dice display, NPC registration, agent rules
5. Check Flask server → restore web UI scene (automatic: `restore_scene`)
6. Display current session settings to the user:
   - `show_dice_result` (true/false)
   - `show_system_log` (true/false)
   - `player_input_mode` (batch/sequential)
   - `sd_illustration` (true/false)

> Showing a blank screen or a previous scene in the web UI is prohibited.

### New Game (turn_count=0, status=active)
game_start.py handles: scenario selection → ruleset selection → character making → data initialization.
After that, the GM delivers the opening narration (GM Turn 0).

1. Check scenario.json's opening.narrative
2. Proceed following the GM turn template:
   - [1a] Set opening direction (first scene, mood, characters appearing)
   - [1b] Call agents (worldbuilding consistency, NPC dialogue if present)
   - [2] Write opening narration (character introduction + scene description)
   - [3] System reflection (gm-update: background illustration + character layers)
3. Output narration → await user action

> Asking "What do you want to do?" without an opening is prohibited. Characters must appear in the world before they can act.

### Load (Saved Game Resume)
Skips scenario/ruleset/character selection — loads previous save data and resumes play.

1. Common Steps 1-5 above
2. Read `entities/{scenario_id}/` (players, npcs, objects)
3. Read `data/scenario.json` + `data/rules.json`
4. Restore current scene via gm-update (background + NPC layers)
5. Present current situation summary to user → await action

### game_start.py Commands
```bash
python game_start.py                    # Interactive: select scenario
python game_start.py new lost_treasure  # New game with specific scenario
python game_start.py continue karendel_journey --from lost_treasure  # Continue
python game_start.py load               # Load save
```
